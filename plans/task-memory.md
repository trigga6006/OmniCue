# Plan: Session Memory + Context Graft

> Persistent conversation-linked work-state memory with diff-aware resume injection.

## Goal

When a user reopens a saved OmniCue conversation, the harness should be able to:

- remember what the user was doing when that conversation last paused
- remember what the agent was working on or waiting for
- compare saved state to the current live desktop state
- inject a compact, deterministic resume block on the first resumed turn

This plan intentionally supports both:

1. reliable resume of saved conversation threads
2. broader queryable work-state memory across conversations

The resume path is the critical path. Cross-session querying is a second layer on top of that.

## Core Corrections To Keep This Sound

The previous direction was strong, but a few things need to match OmniCue's real architecture:

1. `conversationId` is the persisted identity, not `sessionId`.
   `sessionId` in the current app is an ephemeral runtime ID generated by the renderer and replaced when a conversation is loaded. It should never be the primary key for stored memory.

2. Resume state should be stored with, or directly alongside, the persisted conversation.
   OmniCue already has one JSON file per conversation in `src/main/conversations.ts`. The resume capsule should be attached to that conversation record or stored as a clearly linked sidecar keyed by `conversationId`.

3. Resume injection must be provider-aware, but the decision to inject should be provider-agnostic.
   Today only Codex uses `resumeMode` in a meaningful way. The revised implementation should compute the graft once in main and adapt the injection strategy per provider.

4. Event capture should start from explicit conversation events first.
   User-message, assistant-finish, interaction-request, and save-time capture are easy to link correctly to a conversation. Focus-change and periodic capture are valuable, but they require active-conversation tracking and should come after the explicit event path is solid.

5. Query APIs are useful, but they are not the critical path.
   Resume quality should not depend on building a large agent-facing tool surface first.

---

## Design Principles

1. Persistence answers: "What state did we save?"
2. Context graft answers: "How do we re-inject that state on resume?"
3. No LLM is required in the critical path.
4. Resume must degrade gracefully when state is missing or stale.
5. The live desktop always wins over stale saved state.

## Architecture

```text
                   Capture Layer

  renderer send      main stream finish      interaction request
       |                    |                       |
       +----------+---------+-----------+-----------+
                  |                     |
                  v                     v
            session-memory collector (conversation-aware)
                  |
                  +--> append timeline entry
                  |
                  +--> derive latest resume capsule
                              |
                              v
                    persisted with conversation


                   Resume Layer

  loadConversation(conversationId)
          |
          +--> read conversation + resume capsule
          +--> collectSnapshot() for live state
          +--> generateResumeGraft(saved, live)
          +--> inject on first resumed turn


                   Query Layer

  HTTP /session-memory
  IPC  session-memory:query
  optional action wrapper later
```

## Data Model

### 1. Work-State Timeline Entry

File: `src/main/session-memory/types.ts`

```ts
export interface WorkStateEntry {
  id: string
  timestamp: number

  // Stable persisted identity
  conversationId: string

  // Ephemeral runtime identity, useful for debugging but not indexing
  runtimeSessionId?: string

  provider: string

  desktop: {
    activeApp: string
    processName: string
    windowTitle: string
    display?: number
    packId?: string
    packVariant?: string
  }

  context?: {
    browser?: { pageTitle?: string; site?: string; browserFamily?: string }
    editor?: { workspacePath?: string; openFile?: string; language?: string; projectName?: string }
    terminal?: { cwd?: string; shell?: string; isAdmin?: boolean }
    fileExplorer?: { currentPath?: string; folderLabel?: string }
    clipboard?: string
  }

  agent?: {
    lastUserMessage?: string
    lastAssistantMessage?: string
    lastToolUse?: { name: string; input?: string }
    waitingOn?: string
    pendingInteractions?: Array<{ kind: string; title: string }>
  }

  tags: string[]
  summary: string
  trigger: 'user-message' | 'assistant-finish' | 'interaction-request' | 'focus-change' | 'periodic' | 'manual'
}
```

### 2. Resume Capsule

The capsule should be the compact "best current understanding" of that conversation's resumable state.

```ts
export interface ResumeCapsule {
  updatedAt: number
  conversationId: string
  provider: string

  goal?: string
  lastUserMessage?: string
  lastAssistantMessage?: string
  lastAssistantAction?: string

  desktop: {
    activeApp: string
    processName: string
    windowTitle: string
    display?: number
    packId?: string
    packVariant?: string
  }

  context?: {
    browser?: { pageTitle?: string; site?: string; browserFamily?: string }
    editor?: { workspacePath?: string; openFile?: string; language?: string; projectName?: string }
    terminal?: { cwd?: string; shell?: string; isAdmin?: boolean }
    fileExplorer?: { currentPath?: string; folderLabel?: string }
  }

  pending?: {
    waitingOn?: string
    pendingInteractions?: Array<{ kind: string; title: string }>
    lastToolUse?: { name: string; input?: string }
  }

  tags: string[]
  summary: string
}
```

### 3. Conversation Schema Extension

This is the cleanest fit with the existing persistence layer.

File: `src/main/conversations.ts`

Extend `StoredConversation`:

```ts
export interface StoredConversation {
  id: string
  title: string
  provider: string
  createdAt: number
  updatedAt: number
  messageCount: number
  messages: StoredMessage[]
  resumeCapsule?: ResumeCapsule
}
```

This avoids a second persistence tree for the thing that is always loaded together on resume.

### 4. Session-Memory Timeline Index

For broader query support, keep a separate memory timeline store keyed by `conversationId`.

Storage:

```text
userData/
  session-memory/
    index.json
    {conversationId}.json
```

Index shape:

```ts
export interface SessionMemorySummary {
  conversationId: string
  title?: string
  provider?: string
  lastActivity: number
  entryCount: number
  latestSummary: string
  latestDesktopApp: string
  tags: string[]
}
```

## Storage Strategy

### Conversation file

- canonical source for messages
- also stores `resumeCapsule`

### Session-memory timeline

- append-only bounded history for querying past work-state entries
- keyed by `conversationId`

### Limits

- max 100 timeline files
- max 200 entries per conversation
- prune entries older than 7 days
- prune matching timeline when a conversation is deleted

## Phase 1: Explicit Event Capture First

Start with the capture points that already have correct conversation identity.

### Collector

File: `src/main/session-memory/collector.ts`

Public entry points:

```ts
onUserMessage(args: {
  conversationId: string
  runtimeSessionId: string
  provider: string
  message: string
  snapshot: DesktopSnapshot
}): void

onAssistantFinish(args: {
  conversationId: string
  runtimeSessionId: string
  provider: string
  message: string
  lastToolUse?: { name: string; input?: string }
  waitingOn?: string
  snapshot?: DesktopSnapshot
}): void

onInteractionRequest(args: {
  conversationId: string
  runtimeSessionId: string
  provider: string
  interaction: { kind: string; title: string }
  snapshot?: DesktopSnapshot
}): void

captureManual(args: {
  conversationId: string
  runtimeSessionId?: string
  provider: string
  includeClipboard?: boolean
}): Promise<void>
```

### Why explicit events first

These are all places where OmniCue already knows:

- which persisted conversation is active
- which runtime session is active
- which provider is active

That makes the initial implementation correct without guessing.

### Integration points

1. `src/main/ipc.ts`
   Hook right before `streamAiResponse(...)` is called, because `ipc.ts` currently has access to:
   - `payload.sessionId`
   - resolved `cwd`
   - provider
   - the outbound message array

   This is the best place to call `onUserMessage(...)`.

2. `src/main/ipc.ts` stream callbacks
   - `onFinish(...)` -> call `onAssistantFinish(...)`
   - `onToolUse(...)` -> cache the most recent tool use for that runtime session
   - `onInteractionRequest(...)` -> call `onInteractionRequest(...)`

3. `src/main/conversations.ts`
   Do not make this file reach into runtime session state directly.
   Instead, expose a small helper like:

   ```ts
   updateConversationResumeCapsule(conversationId, capsule): void
   ```

   and let the collector or a session-memory service call it.

## Phase 2: Resume Capsule Derivation

The capsule should be rebuilt from the most recent timeline entries whenever a conversation meaningfully changes.

### Capsule builder

File: `src/main/session-memory/capsule.ts`

```ts
buildResumeCapsule(args: {
  conversation: StoredConversation
  recentEntries: WorkStateEntry[]
}): ResumeCapsule
```

### Source of truth rules

1. Prefer the latest timeline entry for desktop/context state.
2. Fall back to the latest stored message metadata if no timeline entry exists.
3. Prefer the latest assistant/tool/interaction-derived state for `pending`.
4. Prefer conversation title or first user message for `goal`.

### Save timing

Update the capsule when:

- assistant finishes a response
- an interaction request is created
- the conversation is explicitly saved
- the conversation is auto-saved after message changes

That keeps resume state close to the actual thread.

## Phase 3: Context Graft

This is the runtime half that turns stored state into useful resume behavior.

### Graft generator

File: `src/main/session-memory/graft.ts`

```ts
export interface ResumeGraft {
  text: string
  hasChanges: boolean
  staleFields: string[]
}

export function generateResumeGraft(
  capsule: ResumeCapsule,
  live: DesktopSnapshot
): ResumeGraft
```

### Diff rules

Compare saved vs live:

- active app
- window title
- pack id / variant
- workspace path or terminal cwd
- open file
- browser page title / site

Also compute elapsed time since save.

### Output shape

The graft should be a compact deterministic block, for example:

```xml
<resume-context>
  <saved-state when="2026-04-11T18:20:00Z" elapsed="2h">
    <goal>Debugging auth middleware</goal>
    <summary>Editing auth/jwt.ts in VS Code</summary>
    <last-user-message>Can you check why refresh token validation fails?</last-user-message>
    <last-assistant-action>Suggested fix in auth/jwt.ts around expiry comparison</last-assistant-action>
    <pending>User was reviewing the suggested fix</pending>
  </saved-state>

  <current-state>
    <summary>Reading JWT Best Practices in Chrome</summary>
  </current-state>

  <changes>
    <change>Active app changed from VS Code to Chrome</change>
    <change>Open file is no longer visible</change>
  </changes>
</resume-context>
```

### Injection strategy

This must be handled in main before provider dispatch, not only inside the Codex path.

#### Add to `sendAiMessage` payload

Extend the renderer -> main payload to include:

```ts
{
  sessionId: string
  conversationId: string
  provider?: string
  resumeMode?: 'normal' | 'replay-seed'
}
```

This is required because `ai.ts` currently receives `sessionId` but not `conversationId`.

#### New preparation step in `src/main/ai.ts`

Add a shared helper before provider routing:

```ts
prepareMessagesForResume(args: {
  messages: ChatMessage[]
  conversationId?: string
  resumeMode: 'normal' | 'replay-seed'
  includeLiveSnapshot: boolean
}): Promise<{
  messages: ChatMessage[]
  graftText?: string
}>
```

Behavior:

- if not resumed, do nothing
- if resumed and capsule exists, collect a live snapshot and generate the graft
- prepend a synthetic `system` message for providers that respect conversation messages
- return `graftText` separately for provider paths that build custom text prompts

#### Provider-specific adaptation

1. Codex app-server path
   In `CodexAppServerClient.streamSession(...)`, prepend `graftText` before the replay seed/current user text.

2. Codex CLI fallback / Claude CLI / OpenCode / Kimi / other CLI-like prompt builders
   The synthetic `system` message will naturally flow through `buildPrompt(...)`.

3. Direct API providers
   The synthetic `system` message should be included in the message array passed to the provider request builder.

This avoids splitting resume logic across unrelated branches.

## Phase 4: Active-Conversation Tracking For Focus/Periodic Capture

Only after explicit event capture and resume graft work well should the harness add passive capture.

### Why this is Phase 4

Right now main does not have a durable notion of "this conversation is actively in focus" outside send/stream events.
That has to be added deliberately.

### Minimal tracking model

File: `src/main/session-memory/runtime.ts`

```ts
export interface ActiveConversationRuntime {
  conversationId: string
  runtimeSessionId: string
  provider: string
  panelVisible: boolean
  lastActivityAt: number
}
```

Update this from renderer/main events:

- panel opened/closed
- conversation loaded
- new conversation started
- send begins
- stream finishes

### Focus-change hook

After active-conversation runtime exists, hook `src/main/context/focus-history.ts`:

```ts
if (activeConversationRuntime) {
  sessionMemoryCollector.onFocusChange(activeConversationRuntime, info)
}
```

### Periodic capture

Only run while:

- a conversation is active
- the panel is open or the thread has had recent activity
- the last periodic capture is older than 60s

Skip if nothing changed.

## Phase 5: Query Surface

Once the timeline is populated, expose it.

### HTTP

`GET /session-memory`

Suggested query params:

- `conversationId`
- `since`
- `tags`
- `app`
- `provider`
- `limit`
- `summaryOnly`

Return shape:

```ts
export interface SessionMemoryResult {
  conversations: Array<{
    conversationId: string
    title?: string
    provider?: string
    lastActivity: number
    entries: WorkStateEntry[] | string[]
  }>
  totalEntries: number
}
```

### IPC

- `session-memory:query`
- `session-memory:capture`
- `session-memory:get-capsule`

### Action wrapper

Optional. Do this only after HTTP and IPC are working.

Reason: OmniCue's existing action system is primarily desktop-operation oriented, and `ActionResult` currently returns string-like detail, not rich structured memory payloads. If an action wrapper is still desired, either:

- return JSON text in `detail`, or
- extend the shared action result schema first

Do not assume a structured memory action fits the current action shape without that decision.

## Summary / Tag Generation

Keep it deterministic.

### Summary examples

- Browser: `Reading '<pageTitle>' on <site>`
- IDE: `Editing <openFile or projectName> in <activeApp>`
- Terminal: `Terminal at <cwd>`
- Explorer: `Browsing <currentPath or folderLabel>`
- Fallback: `Using <activeApp>: <windowTitle>`

### Tags

Always include low-risk structural tags:

- pack id
- provider
- app family

Optionally include content tags:

- browser site hint
- a few filtered keywords from last user message

Do not overfit tag extraction in v1.

## Edge Cases

| Case | Handling |
|---|---|
| conversation loaded but no capsule exists | fall back to existing resume behavior |
| capsule exists but live snapshot fails | inject saved-state-only graft |
| saved workspace path differs from current | mark as stale, do not present saved path as current truth |
| very old capsule | include elapsed age note in graft |
| provider changed since last save | keep saved provider in capsule but use current provider for new execution |
| conversation deleted | delete timeline file too |
| current conversation has never streamed | still allow manual capture and save-time capsule generation from latest user message |
| privacy | clipboard only on manual capture or explicit inclusion |

## Finished State

When this plan is implemented, OmniCue should be able to do the following reliably:

- save a compact resumable state for each conversation
- reopen that conversation later and compare saved state to the live desktop
- inject a deterministic resume block on the first resumed turn
- let the agent continue intelligently instead of starting from scratch
- optionally query broader work-state history across conversations
